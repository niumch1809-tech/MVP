# AGENTS.md

## 项目名称

AI 成本核验平台

## 项目定位

本项目是面向制造业/硬件项目的多供应商 BOM 成本核验平台，核心定位包括：

- 多供应商 BOM 报价对比
- 历史 BOM 价格库沉淀
- 成本异常核验
- AI 分析报告生成

平台不是泛数据分析工具，所有功能都必须围绕 BOM、物料、供应商报价、历史价格和成本核验展开。

## 当前阶段

MVP 半自动版本。

当前阶段优先验证真实业务闭环：

- 能上传真实 BOM/报价文件
- 能解析表格字段
- 能按物料明细展示数据
- 能按产品、供应商、品类、物料维度筛选和追溯
- 能对比多家供应商报价
- 能按供应商查看成本结构占比
- 能接入网页内配置的材料参考价源
- 能识别可追溯的成本异常
- 能生成半自动分析结论

不要为了复杂 UI、复杂 AI 或通用分析能力牺牲 BOM 成本核验主流程。

## 核心业务流程

1. 用户上传当前项目的多家供应商 BOM。
2. 上传时应填写产品名称，后续筛选、对比、导出都必须保留产品维度。
3. 用户可上传历史 BOM，作为材料价格参考库。
4. 系统解析 BOM 字段。
5. 系统统一物料名称、品类、单位。
6. 系统进行供应商报价对比。
7. 系统按供应商展示成本结构占比，不能把多家供应商混在一个饼图中。
8. 系统识别异常价格、异常用量、缺项风险和材料参考价偏离。
9. 系统生成成本分析结论和供应商沟通建议。

## 关键业务认知

- 一款灯具通常由许多配件组成，成本核验必须以 BOM 明细物料为核心。
- 数据看板不能只展示一个总成本项，应支持按物料、品类、供应商、项目维度拆解成本。
- 多供应商对比的关键不是简单比较总价，而是逐项比较物料单价、用量、规格、单位、缺项和历史价格偏差。
- 产品名称是重要业务维度；同名物料在不同产品下不能默认合并，物料对比应至少按 `产品 + 物料名称` 隔离。
- 成本结构占比必须按供应商分别展示，不能把所有供应商金额混入同一个饼图。
- 历史 BOM 是材料价格参考库的来源之一，后续需要沉淀为可复用的历史价格库。
- 材料参考价可以来自网页上传的价格表、网页输入的接口 URL、后端环境变量或 MVP mock，优先级应清晰可追溯。
- 所有异常结论必须能追溯到具体物料行、供应商、字段值和计算规则。

## 数据隐私与 Git 边界

- 供应商 BOM、报价表、解析后的本地数据和材料价格表都可能涉及商业隐私，默认不得提交到 GitHub。
- `.gitignore` 已忽略 `data/*.json`、`uploads/*`、`work/`、`outputs/` 等本地数据目录。
- 允许提交的仅是占位文件，例如 `data/.gitkeep` 和 `uploads/.gitkeep`。
- 后续开发不得为了演示效果把真实 BOM、报价、材料价格、客户名、供应商名硬编码进源码或 README。
- 如需样例数据，应使用脱敏 mock，并明确标注为示例数据。
- 若未来部署线上，上传文件和解析结果应进入私有数据库或私有对象存储，并增加鉴权和访问控制。

## 已知优先问题

- 真实用户 BOM/报价文件可能存在复杂表头、合并单元格、多 sheet、中文编码、空行、备注行、不同字段命名等情况，解析逻辑需要持续增强。
- 当前数据看板仍偏粗，需要升级为 BOM 明细级成本对比，突出多个配件组成整灯成本的结构。
- 真实 BOM 中同一物料可能在不同产品、不同规格或不同单位下出现，后续 AI 匹配前不能随意合并。
- 材料参考价表可能字段不统一，需持续增强价格表字段识别和单位换算能力。
- 后续开发中，如果发现文件无法提取数据，应优先修复解析器和字段映射，而不是在前端用假数据掩盖问题。

## 复杂 BOM 解析器接入原则

复杂 BOM 解析能力必须接入当前平台，而不是另起一个平台、另建一个链接或另做一套输出系统。

- 本地地址仍使用当前项目，例如 `http://localhost:3000`。
- GitHub 仍使用当前仓库。
- 在线版仍使用当前 Sites 项目和当前线上链接。
- 复杂 BOM 解析器只负责把复杂输入表转换为平台内部统一的 `CanonicalBomRow` 数据。
- 复杂 BOM 解析器不得破坏现有标准 BOM/模板 BOM 的上传、解析、对比、校准和导出流程。
- 标准 BOM 解析和复杂 BOM 解析应并行存在：可先通过“复杂 BOM 解析模式”开关接入，稳定后再做自动识别。
- 自动识别时应优先判断工作表结构，例如多区块、重复表头、合并单元格、分段标题、小计/合计行、空行和备注行。
- 复杂 BOM 应先做结构识别，再做字段抽取；不要直接按固定行号或第一行表头硬解析。
- 复杂 BOM 解析应支持多 sheet，每个 sheet 可以代表不同报价对象、型号或供应商视图。
- 复杂 BOM 解析应支持合并单元格向下/向右继承，但必须在原始字段或元数据中保留来源坐标，便于追溯。
- 复杂 BOM 解析应识别并排除小计/合计行，避免把“材料成本合计”“包装成本”“光源部分成本”等汇总行重复计入物料明细。
- 小计/合计、人工/管理/利润、核验总成本/出厂价等汇总信息可以保留为汇总数据，但不能当作普通物料参与明细累计。
- 解析结果应保留原始行号、工作表名、来源区块、原始字段、推断字段、行类型和异常原因。
- 对无法可靠识别的行，不要静默丢弃；应进入解析预览或异常提示，允许用户后续手工校准。
- AI 可以辅助识别区块、字段和物料语义，但最终成本计算必须继续使用可审计的规则逻辑。

复杂 BOM 解析器最终必须输出到现有内部数据结构，并复用现有业务链路：

```text
复杂 BOM / 标准 BOM
  -> 统一 CanonicalBomRow
  -> 手工校准
  -> 报价对比
  -> 数据表与预警
  -> 数据输出
```

输出端必须继续使用既有 BOM 输入输出模板和当前模板导出逻辑，不能因为复杂 BOM 输入而生成另一套格式。导出仍应保持：

- 分类
- 物料
- 规格描述
- 供应商 1 成本
- 供应商 2 成本
- 差价
- 差价 %
- 供应商 1 品类合计
- 供应商 2 品类合计
- 品类差异金额/百分比
- 底部保留材料成本合计、人工/管理/利润合计、核验总成本/出厂价
- 保留既有模板样式、框线、合并单元格、颜色标记和差异标记规则

## 开发原则

- 优先实现真实可用流程，不追求复杂 UI。
- 所有功能要围绕 BOM 成本核验，不做泛数据分析平台。
- AI 功能先预留接口，可先用规则和 mock 数据实现。
- 所有计算逻辑要可追溯，不能只输出黑盒结论。
- 上传、解析、标准化、对比、异常识别、报告生成应形成完整闭环。
- 解析失败时要给出可理解的失败原因和可修正建议。
- 前端展示要服务业务判断，重点突出物料明细、供应商差异、历史价格偏差和风险项。
- 尽量让网页自身完成核价、制表、绘图和导出，不要依赖 Codex 或人工临时计算。
- AI 功能应作为辅助匹配和报告生成，不应替代可审计的规则计算。

## 代码规范

- TypeScript 类型清晰，核心数据结构必须定义在 `src/types/`。
- 成本计算逻辑独立封装，避免散落在页面组件中。
- 表格解析逻辑独立封装，优先放在 `src/lib/bom/`。
- 图表组件应可复用，避免将业务计算写死在图表组件里。
- 异常检测规则要可配置，便于后续增加规则、阈值和行业参数。
- API Route 只做请求处理、校验和调用服务，不堆积复杂业务逻辑。
- UI 组件只负责展示和轻量交互，不承担 BOM 解析或成本核验算法。
- 对外展示的异常结论必须包含来源数据、计算依据和建议动作。

## 成本核验规则方向

后续规则应优先覆盖：

- 同一物料在不同供应商之间的单价差异。
- 同一产品下同一物料在不同供应商之间的单价差异。
- 同一物料相对历史价格库的涨跌幅。
- 同一物料相对网页上传材料价格表或外部价格接口的偏离。
- 同一物料在当前 BOM 与历史 BOM 之间的用量异常。
- 当前报价相对历史 BOM 的缺项风险。
- 同名不同规格、同规格不同单位导致的误判风险。
- 总成本差异的明细拆解，例如结构件、电子料、包装、人工或辅料占比变化。

规则输出应包括：

- 异常类型
- 风险等级
- 物料名称/标准名
- 供应商
- 产品名称
- 当前值
- 对比基准
- 差异比例或差异金额
- 计算规则
- 建议动作

## 后续开发优先级建议

1. 增强真实 Excel/BOM 文件解析能力，包括多 sheet、表头定位、合并单元格、空行过滤和字段映射确认。
2. 接入复杂 BOM 解析器 v1，优先支持多区块识别、重复表头识别、合并单元格继承、小计/合计行排除、规格描述保留和解析结构预览。
3. 复杂 BOM 解析器接入后必须继续输出到现有 `CanonicalBomRow`，并复用当前对比、手工校准、预警和模板导出流程。
4. 增强产品维度和项目维度管理，例如产品列表、历史上传记录、按产品归档。
5. 将看板升级为物料明细级对比，支持按产品、供应商、品类、物料筛选和排序。
6. 建立历史 BOM 价格库雏形，支持当前报价与历史价格对比。
7. 将异常检测规则配置化，支持阈值、单位换算和规则开关。
8. 增加分析报告导出能力。
9. 接入 AI 物料匹配和 AI 报告生成接口。
